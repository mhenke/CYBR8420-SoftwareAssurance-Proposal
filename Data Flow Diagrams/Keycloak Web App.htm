2. Cross Site Request Forgery  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting. The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification:By default, Keycloak has multiple layers of CSRF protection implemented

23. Spoofing the External User External Entity  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:External User may be spoofed by an attacker and this may lead to unauthorized access to Keycloak Web App. Consider using a standard authentication mechanism to identify the external entity.
Justification:Keycloak uses standardized authentication mechanisms and other layers of security controls, including standards-based authentication on all requests. For API access, keycloak supports mutual TLS to authenticate on both the client and server ends during communication. Keycloak also uses digital signature validation and public keys to ensure tokens are issued by a trusted authority. Additionally, Keycloak suports MFA to add an additional security layer, inlcuding time-based one-time passwords, push notifications, or SMS codes. Keycloak controls user sessions and token lifetimes, which ensures spoofed tokens or hijakced sessions are ineffective with detailed logging and monitoring tool integration to detect behavior outside of the norm. Documentation of best practices for Keycloak are also available.

24. Elevation Using Impersonation  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Keycloak Web App may be able to impersonate the context of External User in order to gain additional privilege.
Justification:Keycloak uses explicit role-based controls, secure token management, logging and auditing, and adherence to best practices. Fine-grained RBAC is enforced for all administrative actions and privileges can not be escalated by default in Keycloak. Tokens are cryptographically signed to prevent forgery or tampering without detection. Secure flows using OAuth 2.0 and OpenID Connect Standards are used to prevent token interception or misuse, preventing token-based escalation of privilege. Comprehensive logging and monitoring capabilities are availalbe in Keycloak, allowing outliers to be detected earlier and impact of escalation of privilege attempts to be reduced. Keycloak also supports MFA capabilities for sensitive action execution by administrators and other users as well as allowing for the creation of custom authentication flows and policies for an organizationally-tailored security measure.
